00 The mental model

Five words unlock everything else. Learn these and the commands stop feeling random.

WordPlain-English meaning
Repository (repo)A project folder that git is watching. It remembers every saved version of every file.
CommitA saved snapshot with a short note (“added intro section”). Like a labelled restore point.
RemoteThe copy of your repo that lives on GitHub. Usually called origin.
Push / PullPush = send your commits up to GitHub. Pull = bring GitHub’s commits down to you.
BranchA parallel line of work. Try things safely, then merge back into main.
The one loop to remember edit → add → commit → push. That is 90% of daily git. Everything else is a variation on it.

01 One-time setup

Do this once per computer. It installs git and stamps your name on every commit.

A. Install git

Windows macOS

Windows (PowerShell):

winget install --id Git.Git -e

macOS (Terminal, with Homebrew):

brew install git

Check it worked:

git --version

B. Tell git who you are

git config --global user.name "Your Name" git config --global user.email "you@example.com"
Use your GitHub email Use the same email as your GitHub account so your commits link to your profile. The private username@users.noreply.github.com address works too.

C. Sign in to GitHub

This is the step most guides skip — and the wall most beginners hit. GitHub no longer accepts your account password on the command line, so your first git push will fail until you sign in properly.

No account yet? Create a free one at github.com/signup first — the username and email you pick here are what you configured in step B.

The easiest, official way is GitHub CLI, which works on Windows, macOS, and Linux and stores your login for you:

# install GitHub CLI (pick your system): winget install GitHub.cli # Windows brew install gh # macOS / Linux (Homebrew) gh auth login # then choose: GitHub.com -> HTTPS -> Yes (authenticate Git)
Or just let it prompt you On Windows and macOS, Git ships with Git Credential Manager. The first time you git clone or push a private repo, a browser window opens to sign in — do that once and you’re set. (On macOS, install it with brew install --cask git-credential-manager if needed.)

02 The everyday loop

Get a project, change it, save the change, send it to GitHub. This is the core rhythm.

  1. Clone — download a GitHub repo to your computer.
    git clone https://github.com/you/repo.git
  2. Check status — see what you changed. Run this often.
    git status
  3. Stage (add) — pick which changes go into the next snapshot.
    git add . # everything, or name a file: git add notes.md
  4. Commit — save the snapshot with a short message.
    git commit -m "Describe what changed"
  5. Push — send your commits up to GitHub.
    git push
  6. Pull — before you start, grab anything new from GitHub.
    git pull
Good commit messages Finish the sentence “This commit will…” — e.g. Add contact section, not stuff. Present tense, short, specific.

03 Undo mistakes safely

The reason git is calming: almost nothing is permanent. Pick the tool that matches the mistake.

SituationCommand
Changed a file, want the last saved version back
git restore file.md
Staged a file by mistake (added, not committed)
git restore --staged file.md
Typo in your last commit message
git commit --amend -m "Better message"
A commit you already pushed was wrong
git revert <commit>
Revert vs. delete git revert makes a new commit that cancels a bad one — safe, and it keeps history honest. Avoid rewriting history (reset --hard, force-push) on anything you have shared.

04 Branches & merges

A branch is a safe sandbox. Build your change there, then fold it back into main.

  1. Create and switch to a branch
    git switch -c add-about-page
  2. Do your work — edit, then the usual add and commit.
  3. Switch back to main
    git switch main
  4. Merge your branch in
    git merge add-about-page
Conflicts are normal If git says “merge conflict”, two edits touched the same line. Open the file, keep the version you want (delete the <<<<<<< markers), then git add and git commit. That’s it.

05 Pull requests on GitHub

A pull request (PR) is how you propose a change on GitHub — a place to review before it joins main.

  1. Push your branch to GitHub
    git push -u origin add-about-page
  2. Open the PR — GitHub shows a “Compare & pull request” button. Click it, add a title and a note, and create.
  3. Review — the PR shows exactly what changed, line by line. Others (or you) can comment.
  4. Merge — click Merge pull request. Your change is now in main.
  5. Tidy up — delete the branch (GitHub offers a button), then locally:
    git switch main git pull
Why PRs, even solo? Working alone, a PR still gives you a clean before/after view of your own change — a great habit and a readable history for anyone who visits your repo.

06 Drive git with GitHub Copilot CLI

Once you know the moves, you rarely have to type them. Ask Copilot CLI in plain English and it runs the git for you — asking permission before anything that changes files.

A. Install & start

Install (needs Node.js 22+), then launch inside your project folder:

npm install -g @github/copilot # or: winget install GitHub.Copilot copilot # start it in your repo folder

On first launch it confirms you trust the folder, then run /login to sign in to GitHub.

B. Prompts that map to what you just learned

You already knowJust say to Copilot
add + commit“Commit my changes with a clear message.”
branch“Make a new branch called add-about-page and switch to it.”
push + PR“Push this branch and open a pull request.”
status / explain“What changed since my last commit?”
undo“Undo my last commit but keep the file changes.”
You stay in control Before Copilot runs anything that modifies or executes files, it asks you to approve. Read the command, then choose Yes, or reject and tell it what to do differently. Press Esc to stop it mid-task. Use plan mode (Shift+Tab) to agree an approach first.
Handy in-session commands /diff review your changes · /pr work with pull requests · @file point Copilot at a specific file · /help full command list.

07 The practice lab

Reading is not learning — doing is. Work these exercises in a safe sandbox you cannot break. For each step, try it yourself first, then open the dropdown to check your approach and understand what just happened.

Set up once: clone this repo and move into the practice folder.

git clone https://github.com/eriic-builds/git-learning-lab.git cd git-learning-lab/practice
Your compass Stuck at any point? Run git status. It almost always tells you what state you’re in and what to do next.

Exercise 1 — Your first commit

Goal: turn an edit into a saved snapshot in history.

Try it yourself Open hello.txt, add a line like “Practiced by <name> on <date>”, and save. Now — how do you check what git noticed, then save that change as a commit?
Check your approach

See what changed, stage it, then commit it:

git status # shows hello.txt as "modified" git add hello.txt # stage it for the next snapshot git commit -m "Add my name to hello.txt"

Confirm it landed in history:

git log --oneline
What you should see
Terminal — practice
$ git status On branch main Changes not staged for commit: (use "git add <file>..." to update what will be committed) modified: hello.txt no changes added to commit (use "git add" and/or "git commit -a") $ git add hello.txt $ git commit -m "Add my name to hello.txt" [main 5faa7f4] Add my name to hello.txt 1 file changed, 1 insertion(+) $ git log --oneline 5faa7f4 Add my name to hello.txt ae54cec Initial commit
Why two steps (add, then commit)? Git separates choosing what goes in a snapshot (add, the “staging area”) from saving it (commit). That lets you commit only some of your changes and leave the rest for later. A commit is a permanent, labelled point you can always return to.

Exercise 2 — A branch and a merge

Goal: make a change safely off to the side, then fold it into main.

Try it yourself — part 1 Create a new branch called practice-branch and switch onto it. How?
Check your approach
git switch -c practice-branch
What you should see
Terminal — practice
$ git switch -c practice-branch Switched to a new branch 'practice-branch'
What’s going on? The -c means “create”. You now have a second, independent line of work. Commits you make here do not touch main — that’s what makes a branch a safe place to experiment.
Try it yourself — part 2 Add another line to hello.txt, commit it, then switch back to main. What do you expect hello.txt to look like on main?
Check your approach
git add hello.txt git commit -m "Add a second line on a branch" git switch main
What you should see
Terminal — practice
$ git add hello.txt $ git commit -m "Add a second line on a branch" [practice-branch d3f91e7] Add a second line on a branch 1 file changed, 1 insertion(+) $ git switch main Switched to branch 'main' # open hello.txt now — your second line is not here
Notice the surprise. On main, your second line is gone. It hasn’t disappeared — it lives on practice-branch. Each branch has its own version of the files. Switching branches swaps your working files to match that branch. This is the “aha” moment of branching.
Try it yourself — part 3 Bring the branch’s change into main, then tidy up the branch.
Check your approach
git merge practice-branch # now your line appears on main git branch -d practice-branch
What you should see
Terminal — practice
$ git merge practice-branch Updating 5faa7f4..d3f91e7 Fast-forward hello.txt | 1 + 1 file changed, 1 insertion(+) $ git branch -d practice-branch Deleted branch practice-branch (was d3f91e7).
Why merge, then delete? merge copies the branch’s commits into main, so the work is now part of the main line. Once merged, the branch has done its job — -d safely deletes it (git refuses if it isn’t merged yet, protecting you from losing work).

Exercise 3 — Your first pull request

Goal: propose a change on GitHub and merge it there. (Use a repo you own, or fork this one first.)

Try it yourself Make a branch, commit a small edit, and get it up to GitHub so you can open a pull request. Which commands, in order?
Check your approach
git switch -c my-first-pr git add hello.txt git commit -m "Practice change for a pull request" git push -u origin my-first-pr

Then on GitHub: click Compare & pull request, add a title and note, and Create pull request. Review the diff, then Merge. Finally, update your local main:

git switch main git pull
What you should see
Terminal — practice
$ git push -u origin my-first-pr Enumerating objects: 5, done. Counting objects: 100% (5/5), done. Writing objects: 100% (3/3), 320 bytes, done. Total 3 (delta 1), reused 0 (delta 0) remote: remote: Create a pull request for 'my-first-pr' on GitHub by visiting: remote: https://github.com/you/repo/pull/new/my-first-pr remote: To https://github.com/you/repo.git * [new branch] my-first-pr -> my-first-pr branch 'my-first-pr' set up to track 'origin/my-first-pr'.
What is -u origin doing? It pushes your branch to GitHub (origin) and links your local branch to it, so future git push/pull need no extra arguments. Why a PR at all? A pull request is a reviewable, commentable before/after view of your change — a clean record of what changed and why, even when you’re working solo.

08 A real project workflow

How git and GitHub are actually used to build a project over time. This is GitHub flow — the lightweight, branch-based workflow GitHub itself uses for its docs, policies, and roadmap. It scales from a solo project to a big team.

The shape of it Issue → branch → commits → pull request → review → merge → delete branch. Then repeat for the next piece of work. Your main branch always stays in a working, shippable state; all new work happens on branches.

A. Decide what to work on — an Issue

On GitHub, open an Issue for the task or bug (e.g. “Login times out too fast”). Issues are your to-do list: they give the work a number (like #12) you can link to later, and a place to discuss it.

B. Build it — the branch loop

  1. Start from an up-to-date main
    git switch main git pull
  2. Create a short, descriptive branch — one branch per unrelated change.
    git switch -c fix-login-timeout
  3. Make small, complete commits — ideally each commit is one isolated change, so it’s easy to revert or build on later.
    git add . git commit -m "Increase session timeout to 30 min"
  4. Push to back up your work — the first push sets the branch’s upstream; after that, just git push.
    git push -u origin fix-login-timeout

Keep making, committing, and pushing changes until you’re ready for feedback.

C. Propose it — the pull request

  1. Open a pull request on GitHub. Write a summary of what changed and what problem it solves.
  2. Link the issue so it closes automatically when the PR merges — put a keyword in the PR description:
    Closes #12
  3. Review & checks — reviewers comment on specific lines; any configured status checks (automated tests) run on the PR so you catch errors before merging. Push more commits to respond — the PR updates itself.

D. Ship it — merge & clean up

  1. Merge once approved. Your change lands on main, and the linked issue closes.
  2. Delete the branch (GitHub offers a button). Nothing is lost — the PR and its history stay, and a branch can be restored if needed.
  3. Update your local main for the next piece of work.
    git switch main git pull
On real teams: protected branches Shared repos often protect main so you can’t commit to it directly — a merge requires a pull request, a set number of approvals, and passing status checks. This is why the branch-and-PR habit matters even when you work solo: it’s the same flow the whole team relies on.
Do it with Copilot CLI Once the flow is muscle memory, drive it by asking: “Create a branch for issue 12, commit my changes, push, and open a pull request that closes #12.” Copilot runs each step, asking your approval before it acts.

09 Gotchas & checklist

The small traps that trip up newcomers — and how to sidestep them.

The trapWhat works
“Author identity unknown” on first commitRun the two git config --global user.name/email commands from section 01.
Pushed to the wrong branchCheck with git status before pushing — it names your current branch.
Forgot to pull, now push is rejectedRun git pull first, resolve any conflict, then push again.
Committed a file you didn’t mean toAdd its name to a .gitignore file so git stops tracking new ones.
Asked to log in every push (or push is rejected)You haven’t signed in — do the one-time Sign in to GitHub step in section 01 (GitHub CLI or Git Credential Manager). Passwords don’t work on the command line.
